# EXAMINA Technical Debt Register

| ID | Description | Reason Deferred | Priority | Risk | Target Release |
|----|-------------|-----------------|----------|------|----------------|
| TD-001 | RESOLVED — Branch protection status checks require CI to have run at least once | CI workflow now exists (`.github/workflows/ci.yml`) and reports on every PR; resolved in Phase 1. | LOW | LOW | v0.1.0 |
| TD-002 | With `strict` required status checks enabled, GitHub cannot evaluate a required context for a commit that has never been pushed — so direct pushes to `main` are permanently rejected, not just during bootstrap. All changes to `main`, forever, must land via a PR from a branch (CI runs on the `pull_request` event and reports against the PR's head SHA before merge). This is a standing operational characteristic of the Phase -1 branch protection configuration, not a one-time bootstrap hurdle that "resolves" after the first CI run. | Inherent to how GitHub evaluates required status checks against a specific commit SHA; not fixable without weakening branch protection (removing `strict` or the required contexts), which was explicitly requested in Phase -1. | LOW | LOW | N/A — accepted as the intended workflow |
| TD-003 | `LocalBridgeClient.analyze` and `RemoteBridgeClient.analyze` are Phase 1 stubs — the former returns fixed placeholder data, the latter always raises `BridgeError(code="BRIDGE_UNAVAILABLE")`. Neither invokes real PRISM or real HTTP. | Explicitly out of scope for Phase 1 per the bridge interface prompt; real implementations are blocked on Phase 2's report engine needing real PRISM output to test against. | LOW | LOW | v0.2.0 (Phase 2) |
| TD-004 | `BRIDGE_SPEC_v1.0.md`'s field-level type contract (`BridgeRequest.declared_mime_type`, `BridgeFact.fact_id`-less shape, `BridgeStatus` enum, etc.) was superseded by `BRIDGE_SPEC_v1.1.md` in Phase 1 without ever being implemented — `src/examina/bridge/types.py` was written directly against v1.1. See `docs/adr/ADR-0001-bridge-type-schema-amendment.md`. | The v1.0 field set, if implemented literally as originally frozen, would have shipped `BridgeFact`/`BridgeContradiction`/`BridgeHypothesis` types incompatible with the traceability chain in `EVIDENCE_TRACEABILITY_v1.0.md`; amended via ADR before any code was written against it. | LOW | LOW | N/A — documented, not pending work |
| TD-005 | `HistoryEvent.supporting_signals` is always an empty list in Phase 2's `build_history()` (`src/examina/report/history.py`). `BridgeTimelineEvent` carries no linkage to the `BridgeFact`s that support it, so there is no data available yet to populate per-event signal IDs. | Wiring signal IDs from `EvidenceSection` to individual timeline events requires either a bridge-side linkage field or cross-referencing heuristics that are out of scope for the Phase 2 report engine; deferred until Phase 3 integration work defines how timeline events relate to evidence signals. | LOW | LOW | Phase 3 |
| TD-006 | `tests/security/test_upload_security.py::TestEicarClamAv` (the EICAR-string malware-detection test) is skipped via `pytest.mark.skipif` whenever `clamdscan` is not found on `PATH`. `scan_for_malware()`'s `enforce` branch is otherwise fully covered (100%) by mocked-subprocess tests in `tests/unit/test_clamav_scan.py`, but no test in this repository exercises a real ClamAV daemon end-to-end. | Installing and running a live `clamd` daemon is not available in this development environment or in CI (`ubuntu-latest` runners have no ClamAV installed). Real-signature verification against a live daemon is deliberately left as a manual/staging-environment check. | LOW | LOW | To run the EICAR test: install ClamAV (`apt install clamav-daemon` or equivalent), start `clamd`, ensure `clamdscan` is on `PATH`, then re-run `pytest tests/security/`. |
| TD-007 | `check_archive_bomb()` (`src/examina/pipeline/steps/archive_check.py`) rejects `application/x-tar`, `application/x-bzip2`, and `application/x-7z-compressed` immediately with `DecompressionBombError`, without attempting to inspect their real uncompressed size or compression ratio. | These formats need a vetted extraction/inspection library beyond Python's stdlib (`tarfile` can enumerate members similarly to `zipfile`, but bzip2/7z size inspection needs additional dependencies not yet evaluated against Constitution Principle 8). Conservative outright rejection is the correct default until such a library is vetted. | LOW | LOW | Users must extract/convert these archive formats before uploading; revisit if EXAMINA ever needs to accept them directly. |
| TD-008 | MIME detection (`src/examina/pipeline/steps/mime_check.py`) uses a dependency-free magic-byte signature table instead of `python-magic`/`libmagic`. | `python-magic-bin` was evaluated and works locally on Windows (no deadlock), but ships only as a Windows/macOS binary wheel — incompatible with this repository's `ubuntu-latest` CI (`pip install -e .`). See `docs/adr/ADR-0002-mime-detection-signature-table.md`. The signature table only recognizes the 4 allowed EXAMINA file types plus the 5 archive types checked in `archive_check.py`; it is not a general-purpose MIME sniffer. | LOW | LOW | N/A — documented, not pending work |
| TD-009 | `GET /admin/rules` (`src/examina/api/routes/admin.py`) always returns `{"rules": [], "note": "Rule health tracking available in v0.4.0"}` — no real rule-firing data. | Rule-firing/health data originates from PRISM's own rule engine and is not yet surfaced through the bridge contract (`BRIDGE_SPEC_v1.1.md` has no rule-level field, and `RemoteBridgeClient` is still a Phase 1 stub — see TD-003). Real rule health tracking is blocked on a future bridge amendment plus a live `RemoteBridgeClient` implementation. | LOW | LOW | Target: Phase after beta launch. |
| TD-010 | Rate limiting (`src/examina/api/rate_limit.py`) uses slowapi's in-memory backend, which tracks limits per-process. | Adequate for a single-worker development/beta deployment; a multi-worker production deployment (multiple `uvicorn` workers or horizontally scaled instances) needs a shared backend (Redis) so limits are enforced across workers rather than reset per-process. | LOW | LOW | Revisit before a multi-worker production deployment. |
| TD-011 | `src/examina/api/app.py` uses the deprecated `@app.on_event("startup")` hook (produces a `DeprecationWarning` under FastAPI 0.139) instead of a `lifespan` context manager. | The Phase 4 implementation prompt specified `@app.on_event("startup")` explicitly; functionally correct and fully covered by tests, but FastAPI's newer `lifespan` pattern is the forward-looking replacement. | LOW | LOW | Migrate to `lifespan` in a future FastAPI-upgrade pass. |
