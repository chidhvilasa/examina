# EXAMINA Language Specification

Specification Version: 1.0.0
Status: FROZEN

This document implements Constitution Principle 4 (forbidden language is
absolute) and Principle 12 (explainability before brevity). It defines the
complete allowed and forbidden vocabulary, tone rules, the mandatory
uncertainty rule, and the required format for limitation statements. Every
string rendered to a user passes through `src/examina/language/` (stub
introduced this phase, enforcement logic implemented starting Phase 2).

## Allowed Vocabulary

Findings are described using calibrated, evidence-oriented language:

- **Likelihood language:** "consistent with," "shows signs of," "suggests,"
  "is compatible with," "does not rule out."
- **Confidence language:** "High confidence," "Medium confidence," "Low
  confidence," "insufficient evidence" — always as a labeled level, never
  a bare adjective standing alone without its confidence framing.
- **Artifact-focused subjects:** "this image," "this document," "this
  file" — never a person as the grammatical subject of a finding.
- **Hedged process language:** "appears to have been," "was likely,"
  "may have been" — for reconstructed history (`HistorySection`), never
  stated as settled fact.

## Forbidden Vocabulary (Complete)

The following words, and morphological variants of them (plurals, other
tenses, other parts of speech), never appear in any user-facing EXAMINA
output, at any confidence level, in any section:

| Forbidden | Why | Use instead |
|---|---|---|
| fake / faked / faking | Implies deliberate deception by a person | "shows signs of manipulation," "likely AI-generated" |
| forgery / forged | Legal/criminal implication EXAMINA cannot establish | "inconsistent with an unedited original" |
| fraud / fraudulent | Legal implication of intent | "the evidence does not support authenticity" |
| definitely / definite | False certainty | "High confidence" (never "certain") |
| guaranteed | False certainty | "supported by strong evidence" |
| confirmed / confirms | Implies verification beyond what signal analysis provides | "consistent with," "supported by" |
| proof / proves | EXAMINA produces evidence, never proof | "evidence suggests," "signals indicate" |
| proven | Same as "proof" | "supported by available evidence" |

This table is the enforcement source of truth: the language package's
denylist is generated from it, not the other way around. Adding a word to
this list (via a new specification version) always takes effect
immediately upon the next release; the reverse — removing a word — is
never done without a Constitution amendment, since Principle 4 names all
eight of these words explicitly and permanently.

Additionally forbidden, as instances of Constitution Principle 13
(EXAMINA does not accuse), regardless of whether they appear in the table
above: any construction naming a specific person (sender, uploader,
photographer, source) as the subject of a manipulation or deception claim
— e.g. "the sender fabricated this image" is forbidden even though no
single word in it appears in the table, because the sentence structure
itself violates Principle 13.

## Tone Rules

**Correct:** "This image shows signs consistent with AI generation
(Medium confidence). Frequency-domain analysis and file structure both
point in this direction, though the underlying signals are not fully
conclusive."

**Incorrect:** "This image is fake. Our AI detector confirmed it was
generated by an AI tool."

**Correct:** "No contradictions were found between this document's
declared authorship metadata and its structural signals (High
confidence)."

**Incorrect:** "This document is definitely authentic — we guarantee it
was not tampered with."

**Correct:** "The available signals do not provide enough evidence to
assess this file's authenticity (Insufficient evidence)."

**Incorrect:** "We couldn't figure this one out." (too casual, discards
the structured insufficient-evidence framing; omits confidence label
entirely)

## Uncertainty Mandatory Rule

Every finding-bearing sentence (every `TraceableString` in
`EvidenceSection`, `HistorySection`, and `Assessment.summary`) is paired
with an explicit confidence signal — either the sentence itself names a
confidence level (as in the "Correct" examples above), or it appears
nested under a `Signal.strength` / `HistoryEvent.event_confidence_label`
field that carries that information for it. A finding sentence is never
rendered with confidence information entirely absent from its immediate
context.

## Limitation Statement Format

Whenever a report cannot fully address something a user might reasonably
expect (a missing file type feature, a degraded/partial analysis, an
unsupported signal), the limitation is stated in this fixed three-part
format:

1. **What was not assessed** — plainly named, e.g. "Video content within
   this file was not analyzed."
2. **Why** — one clause, factual, no apology or hedging beyond the fact
   itself, e.g. "EXAMINA v1 supports image and document analysis only."
3. **What this means for the reader** — one sentence connecting the gap
   back to how they should weigh the rest of the report, e.g. "Findings
   below cover the embedded images and document structure only, not any
   video track."

This three-part format is used identically for every limitation, whether
it originates from `FAILURE_SPEC_v1.0.md`'s failure categories or from a
routine "this file type/feature is out of scope" case.
